How this works

SchedulePlanner finds the scheduling and modality bottlenecks that keep students from finishing a 2-year LAMC program in two years, and shows the smallest schedule changes that unblock the most student paths.

1 · What this app does

You give it the class schedule and the official program requirements. It builds a term-by-term plan for two student archetypes (full-time and part-time), checks whether each can finish on the official map, and flags the courses whose supply — when, how often, and how they're offered — blocks completion. The schedule itself is produced by a deterministic solver, never by AI.

2 · The two ways to use it

Option 1 — Use a workbook. Analyze an Excel .xlsx you provide (sheets sections, catalog, programs), or click Load demo data to run the bundled sample with no setup. Pick this when you already have an export.

Option 2 — Build from live LACCD data. Enter a Campus, one or more Terms, and a Program; the app pulls the published schedule and the program's default pathway straight from LACCD, builds the workbook for you, and analyzes it. Pick this when you have no file. One program per run.

Both paths end in the same analysis — Option 2 just builds the workbook for you and adds a live-data reconciliation panel.

3 · The inputs (Option 2)

  • Campus — the LACCD college code, e.g. LAMC (Los Angeles Mission College).
  • Terms — LACCD term codes (the same codes on the public schedule), comma-separated, e.g. 2264,2266,2268. The app reads the season from each code: one ending in 8 is Fall, otherwise Spring.
  • Program — a program name as it appears in LACCD Program Mapper, e.g. Biology.

4 · Reading the results

Each program shows a status badge and two cohorts (student archetypes):

  • Full-time / Part-time — each shows how many terms it takes to finish (≈ years), and the per-term plan (T1, T2, … list the courses in each term).
  • Status badge — on track (completes on the official map), map broken (the official map has problems), or needs fix (the full-time path needs an added offering to finish on time).
  • fix: — when a path needs help, the suggested minimum change, e.g. add a course in a given season.

5 · Supply diagnostics

These flag required courses whose offering pattern blocks completion:

  • Rotation gaps — a required course offered in too few of the terms analyzed (shown as offered / total).
  • Single-section risk — only one section exists, so a single cancellation or conflict blocks everyone.
  • Capacity / fill-rate (modality mismatch) — a required course whose seats aren't filling (a low enrolled ÷ capacity %). Needs real enrollment counts, so on live data it's an experimental add-on (below).
  • Under-supply — demand exceeds the seats offered. On live data this shows as how many of a course's sections are waitlisted (at capacity); with the enrollment export it sharpens to the exact waitlist headcount.
  • Time conflicts — two (or more) required courses whose meeting times collide, so a student can't take them together and falls a term behind. Computed live from each section's days/times: flags pairs whose every section overlaps, plus terms whose required load has no conflict-free combination.
  • Off-grid sections — a section whose start time isn't on the college's standardized time-block grid (e.g. a 9:00 start where the grid is 8:55), checked against the published Fall / Spring / Winter / Summer time blocks. Off-grid starts are a common source of avoidable conflicts.

6 · Live-data extras (Option 2)

Reconciliation shows how many of the program's required courses were actually offered in the fetched terms versus not offered (with the missing course codes).

Under-supply runs live: the published schedule marks each section Open / Waitlist / Closed, so a course with waitlisted sections is flagged straight from live data (no export needed) — shown as "2/6 sections waitlisted".

Two experimental add-ons on the live form: the schedule API has no enrolled ÷ capacity counts and the program map has no prerequisites, so two things need extra data — and you can supply both right on the live form. Add enrollment export joins an IR / PeopleSoft .xlsx by term + CRN to unlock the capacity / fill-rate diagnostic (and sharpens under-supply from waitlisted-section breadth to an exact headcount). Include prerequisites from eLumen pulls real prerequisites so the solver can order courses. Both are experimental: the eLumen endpoint's terms-of-use review is pending, and the live↔enrollment join isn't validated on real data, so a real export may match zero sections (you'll see the join count). An empty result means "not yet measurable," not "all clear." For fully validated capacity analysis, use Option 1 with a workbook that already includes the counts.

Transfer GE goal: pick Cal-GETC, IGETC, or CSU GE to add lower-division general education to the plan. Each area's eligible courses come live from ASSIST; when an area has one clear (or recommended) offered course the app schedules it, otherwise it reserves a slot ("choose one Area X course"). A major course that also satisfies a GE area is counted once. GE adds ~20–30 units, so the full-time plan may not fit four terms — the part-time plan will. Best-effort: ASSIST's terms-of-use review is pending.

7 · The Gemma 4 badge

The pill in the top-right shows whether a local Gemma 4 model (via Ollama) is available. When ready, it parses messy free-text prerequisites and writes the plain-language admin briefing. When offline, the app falls back to a rule-based parser and a templated summary. Either way the schedule is always produced by the deterministic solver — the AI never decides it.

8 · Data & privacy

Live data comes from LACCD's public class-schedule API and Program Mapper. The analysis is section- and aggregate-level only — no student-level records are used.

9 · Glossary

Section
One specific offering of a course in a term (its own time, room, and modality).
Cohort
A student archetype the plan is built for — here, full-time and part-time.
Prerequisite
A course that must be completed before another can be taken.
Modality
How a class is delivered — in person, online, or hybrid.
Waitlist
Students queued for a section that is already full — a demand signal.
Term code
LACCD's numeric code for a term (e.g. 2268); the app reads the season from it.
Program Mapper
LACCD's source for a program's required courses and default term-by-term pathway.
IR / PeopleSoft export
The institutional enrollment data (caps, enrolled, waitlist) the demand-side detectors need.
